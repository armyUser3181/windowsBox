# EventCaller 구조 개선안 보고서

## 1. 배경

현재 구조에서 `EventCaller`와 `EventInstance`의 관계가 애매합니다.

- `EventCaller`는 이벤트를 감지하는 객체로서의 역할이 분명합니다.
- `EventInstance`는 실제 실행 로직을 담당하는 객체로서의 역할이 분명합니다.
- 하지만 두 객체가 서로를 직접 관리하는 방식으로 설계되면, 초기 설정이 번거롭고 책임이 분산되어 보입니다.

특히 다음 두 가지 방향이 모두 불편하게 느껴집니다.

1. `EventInstance`가 `EventCaller`를 내부적으로 소유하는 방식
   - 구조가 단순하지만, 실행 단위와 이벤트 수신 단위가 너무 강하게 결합됩니다.

2. `EventCaller`가 `EventInstance`를 완전히 독립적으로 관리하는 방식
   - 책임이 더 명확하지만, 연결 관계를 외부에서 매번 관리해야 하므로 설정 비용이 높아집니다.

이 때문에 중간 지점을 찾는 설계가 필요합니다.

---

## 2. 목표

이 설계의 목표는 다음 세 가지입니다.

- `EventCaller`는 이벤트 수신 책임만 담당한다.
- `EventInstance`는 실행 로직 책임만 담당한다.
- 두 객체 사이의 연결은 별도의 관리 계층이 맡아서 처리한다.

즉, “완전한 내부 소유”는 피하고, “완전한 외부 분리”도 피하는 방식입니다.

---

## 3. 제안하는 중간 지점 설계

### 3.1 핵심 아이디어

`EventCaller`와 `EventInstance`는 직접 소유 관계를 가지지 않되,
`EventRepository` 또는 별도 연결 관리자에 의해서 관계가 등록되고 관리되도록 합니다.

이 방식은 다음과 같이 이해하면 됩니다.

- `EventCaller`: “무슨 이벤트를 받을지”만 결정
- `EventInstance`: “무슨 동작을 실행할지”만 결정
- `EventRepository`: “누가 어떤 인스턴스를 실행할지”를 연결

---

## 4. 역할 분담안

### 4.1 EventCaller

`EventCaller`는 다음 역할만 담당합니다.

- 이벤트 타입을 가진다. 예: `click`, `keydown`
- 대상 요소를 가진다.
- 이벤트가 발생하면 repository에 전달한다.
- 자신이 어떤 인스턴스와 연결되어 있는지는 직접 관리하지 않는다.

즉, `EventCaller`는 “입구”에 가깝습니다.

### 4.2 EventInstance

`EventInstance`는 다음 역할만 담당합니다.

- 실행 로직을 정의한다.
- `resolve`를 통해 이벤트 처리 흐름을 수행한다.
- 자신이 어떤 호출자와 연결되어 있는지는 외부에서 관리한다.

즉, `EventInstance`는 “실행 엔진”에 가깝습니다.

### 4.3 EventRepository

`EventRepository`는 중간 관리자 역할을 맡습니다.

- `EventCaller`를 등록한다.
- `EventInstance`를 등록한다.
- 두 객체 사이의 연결 관계를 저장한다.
- 이벤트가 발생하면 적절한 `EventInstance`로 전달한다.

이 구조에서는 `EventRepository`가 연결의 중심이 됩니다.

---

## 5. 권장 구조 예시

```text
EventCaller -> EventRepository -> EventInstance
```

구체적으로는 다음 흐름이 됩니다.

1. 개발자가 `EventCaller`를 생성한다.
2. 개발자가 `EventInstance`를 생성한다.
3. 둘을 `EventRepository`에 등록한다.
4. 이벤트가 발생하면 `EventCaller`가 repository로 신호를 보낸다.
5. repository가 연결된 `EventInstance`를 찾아 실행한다.

이 구조는 “이벤트 수신”과 “실행 로직”을 분리하면서도, 연결 관계를 한 곳에서 관리할 수 있게 해줍니다.

---

## 6. 연결 관리 방식

### 옵션 A: 단순 연결 맵 방식

가장 단순한 방법입니다.

- `callerId -> instanceId` 형태의 맵을 사용
- repository가 직접 연결 정보를 관리

장점:
- 구현이 쉽다.
- 현재 프로젝트 규모에 매우 잘 맞는다.

단점:
- 연결 정보가 단순해져서 확장성이 제한될 수 있다.

### 옵션 B: 연결 객체 방식

연결 정보를 별도의 객체로 분리합니다.

예:

- `EventBinding`
- `caller`
- `instance`
- `condition`
- `priority`

장점:
- 확장성 좋음
- 나중에 조건부 연결이나 복수 연결이 쉬움

단점:
- 구조가 조금 무거워진다.

### 권장안

현재 프로젝트 기준으로는 “옵션 A”가 가장 현실적입니다.

즉, `EventRepository`가 맵 형태로 연결을 관리하는 방식이 가장 적합합니다.

---

## 7. 장점

### 7.1 책임 분리가 명확함

- `EventCaller`는 이벤트 수신만 담당
- `EventInstance`는 실행만 담당
- 연결 관리자는 연결만 담당

이렇게 나누면 코드의 역할이 훨씬 읽기 쉽습니다.

### 7.2 초기 구조가 과하게 복잡해지지 않음

완전한 분리 방식처럼 `EventCaller`와 `EventInstance`가 서로 완전히 독립적으로 관리되는 구조보다 훨씬 구현 부담이 적습니다.

### 7.3 확장성이 있음

나중에 다음과 같은 기능을 넣기 쉽습니다.

- 하나의 `EventCaller`가 여러 인스턴스를 실행
- 하나의 `EventInstance`가 여러 `EventCaller`에 연결
- 특정 조건에 따라 인스턴스 분기
- 이벤트 우선순위 처리

---

## 8. 단점과 주의점

### 8.1 연결 정보가 분산되지 않도록 주의해야 함

이 구조가 좋으려면 연결 관계를 `EventRepository` 한곳에서 관리해야 합니다.

만약 `EventCaller`와 `EventInstance`가 각각 자기만의 연결 정보를 들고 있으면 오히려 구조가 더 복잡해집니다.

### 8.2 중첩 호출을 방지해야 함

이 구조를 도입하더라도, 이벤트 처리 과정에서 재진입이 발생하면 순환 호출이 생길 수 있습니다.

따라서 다음 원칙이 필요합니다.

- 이벤트 처리 중에 같은 인스턴스를 다시 호출하지 않음
- repository에서 현재 처리 중인 이벤트를 추적함
- 재진입 방지 플래그를 둘 수 있음

### 8.3 초기 설정이 조금 더 필요함

이전 구조보다 선언 단계가 늘어납니다.

즉, “캘러 선언”과 “인스턴스 선언” 외에 “연결 등록”까지 해야 하므로 초기 세팅이 조금 더 늘어납니다.

하지만 그 비용은 장기적인 유지보수 편의성에 비하면 충분히 감당할 만합니다.

---

## 9. 추천안 요약

현재 프로젝트 규모와 목표에 맞는 가장 균형 있는 방식은 다음과 같습니다.

- `EventCaller`는 이벤트 입력만 담당한다.
- `EventInstance`는 행동 실행만 담당한다.
- `EventRepository`가 두 객체를 연결하고 이벤트를 라우팅한다.

즉, “관계는 직접 소유하지 않고, 중앙에서 관리한다”는 방향이 가장 적절합니다.

이 방식은 다음과 같은 이유로 추천됩니다.

1. 현재 구조를 크게 뒤집지 않으면서도 책임을 분리할 수 있다.
2. 향후 기능 확장에 유연하다.
3. 코드의 읽기 쉬움과 유지보수성 사이의 균형이 좋다.

---

## 10. 결론

`EventCaller`를 `EventInstance` 내부에 두는 방식은 단순하지만, 책임이 너무 강하게 결합됩니다.

반대로 완전히 분리하는 방식은 유연하지만, 초기 설정과 관리 부담이 큽니다.

따라서 가장 좋은 중간 지점은 다음입니다.

> `EventCaller`와 `EventInstance`는 서로 직접 소유하지 않고, `EventRepository`가 연결을 관리하는 구조

이 방식은 현재 프로젝트의 규모와 목표에 비추어 볼 때, 가장 현실적이고 안정적인 선택이라고 판단됩니다.
